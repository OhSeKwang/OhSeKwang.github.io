1. HTML5의 시작

HTML5의 시작 : HTML5란?


  • HTML5는 HTML 4.01과 XHTML 1.0, XHTML 1.1을 계승하는 차세대 웹 기술로 HTML 마크업이 1990년대 초 인터넷에 소개된 이래로 계속적으로 진화를 거듭해 왔다.
  • 기존에 존재하던 HTML의 한계를 극복하고, 기존 기술들의 설계를 개선하기 위해 W3C에서 HTML5라는 새로운 기술이 정의되었다.
  • 마이크로소프트, 모질라, 애플, 구글, 오페라 등 모든 웹브라우저 벤더가 참여하고 있는 산업 표준이기도 하며, 현재 파이어폭스, 크롬, 오페라, 사파리 등 브라우저의 최신 버전에서 많은 HTML5 태그가 지원되고 있다.

HTML5의 시작 : HTML5란?


시맨틱 태그 지원

  • HTML5에 시맨틱 태그가 도입됨으로써 기존 (div) 태그를 사용하여 페이지를 구조화 하던 대신 문서 구조에 적합하게 header, footer, nav, section 같은 구조화 마크업을 사용하여 웹 페이지를 보다 잘 설계할 수 있도록 하였다.
  • 웹 개발자들이 구조적으로 웹 페이지를 재사용 가능하게 만들어준다던지, 더 나아가 보이는 화면뿐만 아니라 그 화면을 구성하는데 사용되는 소스 레벨을 재사용할 수 있게 해준다.
  • 그림은 HTML5 웹 페이지를 구성하는 영역을
    표현한 것이다. 기존 HTML4에서는 각 영역들을
    (div id="footer")와 같이 일일이 (div) 태그에
    id를 부여하여 구현하였지만, HTML5에서는
    시맨틱 태그 자체로 정의하여 사용한다.

HTML5의 시작 : HTML5란?


HTML5 웹 페이지를 구성하는 각 영역 요소

  • (header) 요소는 웹 페이지의 머리글을 정의하며, 전체 페이지에서도 사용되지만 일부 (article),(aside) 태그의 머리말로도 정의 가능하다.
  • (nav) 요소는 웹 페이지의 탐색 링크를 위해 문서 전체 혹은 섹션의 네비게이션 영역을 표현하는 웹 페이지의 기본 탐색 링크를 위한 영역이다.
  • (section) 요소는 실제 문서 내용을 담은 문서의 주요 영역을 정의한다.
  • (article) 요소는 문서에서 독립적으로 하나의 컨텐츠로 구분되어 독립적으로 배치된 내용을 담는 영역이다.
  • (aside) 요소는 주로 페이지 내용보다는 관련된 링크, 광고 배너와 같은 보조 컨텐츠를 위해 배치된 영역이다.
  • (footer) 요소는 웹 페이지의 바닥글을 정의한다.

HTML5의 시작 : HTML5란?


웹 폼 기능 지원

  • HTML5에서는 사용자의 입력 정보를 받기 위한 보다 손쉬운 웹 폼 기능을 지원한다.
  • 웹 폼 기능은 기존 HTML4에 이미 존재했던 (input) 태그를 통해 구현할 수 있는데, HTML5에서는 여기에 각종 타입 속성들을 새로 추가하여 range, number, date, search, tel, url, color 등의 기능을 지원한다.
  • 현재 (input) 태그의 속성으로 총 12가지 형태의 각종 타입이 제공되고 있으며, 개발자는 해당 기능을 속성으로 정의하기만 하면 브라우저는 기능별 데이터 타입의 유효성까지 검증 해준다.

HTML5의 시작 : HTML5란?


리치 웹 미디어 기술 지원

  • HTML5에서는 리치 웹 미디어 기술을 위해 캔버스 및 SVG, Video 등의 태그를 제공한다.
  • 캔버스 태그는 웹에서 2차원 그래픽을 그리기 위한 도구를 제공하며, 도형을 회전 및 변환하기도 하고 그라디언트 등 각종 효과를 적용할 수도 있다.
  • 벡터 기반의 Scalable한 그래픽을 지원하는 SVG 태그 및 내장 비디오 및 오디오를 재생하는 Video 태그를 통해 리치한 웹 미디어 페이지를 손쉽게 구현 가능하다.

HTML5에서 리치한 웹 페이지를 위해 지원되는 주요 특징

  • 웹 애플리케이션이 독립적으로 특정 프로토콜 및 미디어 형식을 등록할 수 있는 API
  • 드래그 앤 드롭 제스쳐를 지원할 수 있는 API

HTML5의 시작 : HTML5의 역사와 흐름

  • 웹 기술은 쉬운 마크업 언어를 사용하여 정보를 표현하고 다양한 디바이스의 브라우저에서 데이터를 효과적으로 풍부한 UX 기반으로 활용할 수 있다는 점에서 계속적으로 발전되어 왔다.
  • HTML은 1993년 HTML 1.0이 팀 버너스리(Tim Berners-Lee)에 의해 고안되어 발표된 이후 해를 거듭하며 HTML은 발전에 발전을 거듭하였다. 그리고 1997년 HTML 3.2 및 HTML 4.0을 개발하면서 W3C에서 HTML의 버전을 관리하기 시작하였다.
  • XHTML은 HTML이 갖는 단순함이라는 단점을 극복하기 위해 나온 표준임에도 불구하고 오히려 너무 복잡하다는 평을 받았다. 이는 XML 기반의 제한된 문서 구조를 개발자들에게 요구했기 때문이다.

HTML5의 시작 : HTML5의 역사와 흐름

  • WHATWG는 Apple, Mozilla, Opera가 공동으로 창립한 단체로, 다양한 웹 어플리케이션에 효과적으로 사용할 수 있고 보다 단순하면서도 다양한 확장성을 갖는 HTML5의 스펙을 정의하는 워킹그룹이다.
  • 2007년 W3C에서 구성한 HTML 워킹그룹과 공동으로 지금의 HTML5의 규격을 정의하기 시작하였으며 2008년 1월 그 작업 초안을 발표하기에 이르렀다.
  • HTML5의 주요 기능들은 이미 세계 주요 IT 기업들이 발 빠른 적용을 준비하고 있다. 구글은 최근까지 주력하여 개발하였던 구글 기어스(Google Gears) 대신 HTML5에 집중하고 있으며, 유튜브의 경우에도 기존의 방식인 플래시 플레이어를 사용하지 않고 내장 비디오 태그를 사용하여 동영상을 재생하도록 하고 있다.

HTML5의 시작 : HTML5의 역사와 흐름

  • HTML5에서 새로 정의하는 기능들은 모바일 환경에서 요구하는 웹 앱(Web App)의 요구사항에 적합하며, 앞으로도 관련 표준들이 추가적으로 보강될 예정이다.
  • 웹 앱은 안드로이드, 아이폰과 같이 각기 다른 모바일 OS에 대해 대응하는 네이티브 앱을 개발하지 않고 대신 HTML5의 웹 지원 기능을 이용하여 여러 가지 디바이스를 지원할 수 있어 기존 네이티브 앱(Native App)에 비해 훨씬 비용을 절감할 수 있는 효율적인 접근방법을 제공한다.
  • 웹 앱이 향후 멀티 플랫폼을 대처할 수 있는 중요한 솔루션이라는 데는 이의가 없다. 따라서 HTML5를 공부하면서 웹 앱을 개발하는 것을 함께 공부해보는 것도 최신 기술을 대응하는 좋은 방법이다.

기존 HTML과 HTML5의 차이점

  • HTML5를 간략히 요약해 보면 다음의 그림과 같다.
  • 우선 HTML5의 영역 안에 HTML5가 지원하는 기능, 특성들이 어떠한 것들이 있는지를 보여주고 그 외부에는 HTML5를 지원하는 최신 버전의 브라우저들과 아이폰 웹 애플리케이션들이 여기서는 제시되고 있다.

기존 HTML과 HTML5의 차이점

  • HTML5를 지금까지 사용된 HTML4와 비교하여 정리하면, 다음의 그림에서 나타낸 것과 같은 부분들이 주로 변경되었다.

기존 HTML과 HTML5의 차이점

  • 변경된 부분을 좀 더 살펴보면 첫 번째로 표준 코덱(Codec) 기반의 비디오, 오디오 재생 환경을 내장하고 있다.
  • 기존에는 멀티미디어 재생을 위해서 외부의 플러그-인(Plug-in)을 설치해 재생해야만 했다.
  • HTML5에서는 표준 코덱을 선정해서 브라우저 내에 플레이어를 기본 내장하고 있기 때문에 환경이 많이 개선되었다.
  • 각 브라우저마다 지원하는 바가 다음과 같이 각각 다를 수 있다.

기존 HTML과 HTML5의 차이점

  • HTML5는 그래픽 지원을 위한 캔버스(Canvas)를 제공한다.
  • HTML5에서는 그리기 영역과 그리기 API를 제공하기 때문에 브라우저에서 다양한 그래픽 구현이 가능해졌다.
  • 인터넷 익스플로러를 제외하고 대부분의 브라우저에서는 2D 캔버스는 지원을 하고 있지만, OpenGL ES 2.0 기반의 3D 캔버스의 경우에는 최근에 지원이 발표되고 있다.

기존 HTML과 HTML5의 차이점

  • 로컬에서의 저장 기능과 DB가 지원된다. HTML5에 이르러서야 로컬 저장 기능이 표준화되었다.
  • 인터넷 익스플로러에서의 'userData', 파이어폭스의 'Dom storage', 구글의 기어스(Gears)1)가 이러한 기능을 제공하였지만, 이는 HTML5에서 하나의 표준으로 대치되어 제공된다.

  • 위치 정보 관련 API가 제공된다는 것이다.

기존 HTML과 HTML5의 차이점

  • HTML5에서는 웹 워커(Web Worker)라는 기능을 통해 백그라운드 프로세스를 지원할 수 있다.
  • HTML5에 오면서 웹 워커라는 기능을 도입하여 멀티 프로세스 지원이 가능하게 되었다.
  • 단일시간에서의 다양한 일에 대한 효율적인 처리가 가능해졌기 때문에 시각적으로 빠르게 반응이 이루어질 수 있도록 하는 기능 제공이 가능해졌다.

HTML5의 기본 규칙

  • HTML5는 기존의 HTML와 동일한 마크업 언어이다. 따라서 기본 골격은 기존 HTML과 다르지 않다.
  • 짝이 있는 태그는 항상 쌍으로 존재해야 하며, 열리고 닫히는 태그의 순서가 순서대로 이루어져야 한다.
  • 독타입(DOCTYPE)은 빠지지 않고 반드시 넣어주어야 하며, 태그 이름은 대소문자를 가리지 않는다.
  • HTML5에서 무엇보다 중요시 봐야 할 것은 문서가 구조를 중요시 한다는 것이다. 이를 고려하여 트리 구조를 사용하고 있다.

HTML5 호환성 및 사용 가능한 형식

HTML5 호환성

  • HTML5 언어는 HTML4와 XHTML1 문법과 호환되지만, HTML4 내의 일부 SGML(Standard Generalized Markup Language) 문법과는 호환되지 않는 특성을 가지고 있다.
  • 호환되지 않는 문법으로는 처리 명령어(Processing instructions)와 약식 마크업(Shorthand markup)이 있다.

HTML5 호환성 및 사용 가능한 형식

XML 형식

  • XML은 1996년 W3C에서 제안한 것으로서, 웹상에서 구조화된 문서를 전송 가능하도록 설계된 표준화된 텍스트 형식이다.
  • 인터넷에서 기존에 사용하던 HTML의 한계를 극복하고 SGML의 복잡함을 해결하는 방안으로 제안되었다.
  • XML은 현재 W3C로부터 웹을 좀 더 다양한 목적으로 이용할 수 있도록 하기 위한 도구로서 공식 추천되고 있다.
  • HTML5는 HTML과 XML을 모두 사용해서 작성될 수 있다.

HTML5 호환성 및 사용 가능한 형식

문자 인코딩

  • 문자 인코딩은 문자들의 집합을 부호화하는 방법이다. 웹 페이지를 구성하는 언어가 국가별로 다르기 때문에 각 국가의 언어별로 맞는 문자 인코딩 방식을 지정해야만 화면이 깨짐을 방지할 수 있다.
  • HTML5 웹 페이지를 제작하기 위해서는 문자 인코딩에 관련해서 다음의 세 가지 중 하나가 설정되어야 한다.
  • 전송 단계 : HTTP 헤더에 'Content-Type' 표시
  • 문서 앞부분 : 특수 유니코드 BOM(Byte Order Mark) 문자 표시(인코딩 방법에 대한 정보를 제공)
  • 파일 초기(512 바이트 크기 내) : 메타(Meta) 요소 내 'charset' 속성에 표시

DOM의 정의

  • W3C에 의해 개발되고 있는 프로그래밍 인터페이스 규격인 DOM은 문서의 스타일, 구조, 컨텐츠를 동적으로 액세스하거나 업데이트하도록 하기 위한 플랫폼이자 인터페이스이다.
  • 문서에 대한 구조적인 정보를 제공하고, 문서 구조나 스타일 및 내용을 변경할 수 있도록 프로그램에서 접근할 수 있는 방법을 제공하기 때문에 필수적이다.
  • DOM은 프로그래머가 HTML 페이지나 XML 문서들을 프로그램 객체로 만들거나 수정할 수 있도록 해준다.

HTML5 템플릿

  • HTML5가 W3C에서는 아직도 논의가 계속해서 진행되고 있지만, 개발자의 활동이 활성화되면서 많은 웹 사이트에 적용되고 있다.
  • HTML5가 기존의 HTML 언어를 발전시켜서 플래시와 같은 외부의 플러그인의 도움 없이도 서비스를 제공할 수 있다는 장점을 가지고 있고, 스크립트를 대체시킬만한 다양한 마크-업(Mark-up)인 태그를 사용하게 됨으로써 복잡한 구현 이슈를 많이 최소화시켰기 때문이다.
  • HTML5는 디바이스에 독립적이기 때문에 요즘과 같이 다양한 플랫폼이 존재할 때 웹 어플리케이션 개발 시 매우 유용한 도구로 사용된다.

HTML5 템플릿

  • HTML5의 시대가 열리는데 최고의 기여를 한 애플의 최고경영자(CEO) 스티브 잡스는 이미 이전에 아이폰과 아이패드에서 '플래시'를 사용하지 않을 것이라고 밝혔고, 이를 통해서 HTML5에 대한 관심이 급격히 증가하게 되었다.

드래그(Drag) & 드롭(Drop) API

    기본적인 사용법

  • HTML5에는 드래그 & 드롭을 위한 전용 API가 제공된다. W3C 스펙에서는 드래그 동작을 포인팅 디바이스에서 일련의 'mousemove' 이벤트가 발생 후 'mousedown' 이벤트가 발생함을 의미한다.
  • 드롭 동작은 손가락으로 마우스에서 누르고 있던 것을 해제하는, 즉 손가락을 떼는 동작에 해당된다. 만일 포인팅 디바이스가 없다면, 이 동작은 앞서 설명했던 동작과 유사한 동작이 다른 매체를 통해서 일어나면 된다.
  • 드래그 & 드롭 API에서의 이벤트 발생은 기존 윈도우 응용 프로그램의 드래그 & 드롭 이벤트와 유사하다.
  • 과거처럼 자바스크립트의 마우스 관련 이벤트와는 전혀 상관이 없으며 드래그 대상, 드롭 대상, 이동 콘텐트 등의 정보를 기반으로 동작한다.

드래그(Drag) & 드롭(Drop) API

    기본적인 사용법

  • 드래그 & 드롭 동작인 누르고 어떤 위치로 끌고 가서 다시 떼는 것은 물리적인 노력이 필요하기 때문에 드래그하는 동안에는 빠르게 움직이기 어렵지만 구글의 웹사이트 서비스 중 일부는 이러한 드래그 & 드롭 기능을 적용하여 사용자의 편의성을 극대화하고 있다.

드래그 & 드롭 API 요소

    드래그 대상(Source)

  • 드래그 대상은 끌어올 대상을 의미한다. HTML 요소에 'draggable' 속성 값을 'true'로 설정하면 드래그 대상이 될 수 있다.

    • 드롭 대상(Target)

  • 드롭 대상은 포인트를 끌어다가 어느 한 곳에 놓을 때 그 대상이 되는 위치를 의미한다.

    • 드래그 데이터(Data)

  • 드래그 데이터는 이동하는 대상 컨텐츠에 해당한다. 이동할 데이터는 'DataTransfer' 객체를 통해 이루어지며 드래그 대상에서 'DataTransfer' 객체에 값을 저장하고 드롭 대상에서 'DataTransfer' 객체의 값을 가져온다.

드래그 & 드롭 API 요소

    드롭 이벤트들

  • 드래그 앤 드롭과 관련해 총 7가지 이벤트가 제공된다. 7가지 이벤트들은 드래그 앤 드롭 과정에서 대부분 순차적으로 발생하며, 각 이벤트에 맞는 이벤트 처리가 반드시 필요하다.
  • dragstart 이벤트 : 드래그가 시작될 때 발생한다.
  • dragenter 이벤트 : 이동할 데이터를 마우스로 끌어서 드롭 대상에 들어오는 순간 발생한다.
  • dragover 이벤트 : 이동할 데이터를 마우스로 끌어서 드롭 대상에 올려놓는 동안 계속해서 발생한다.
  • drop 이벤트 : 마우스로 끌어서 놓는 순간 발생한다.
  • drag 이벤트 : 드래그 & 드롭 동작을 지속한다.
  • dragleave 이벤트 : 드래그 & 드롭 동작을 종료한다.
  • dragend 이벤트 : 드래그 & 드롭 동작을 종료한다.

어플리케이션 캐시(Application Cache)

  • 캐시란 사용자가 한참동안 크롬 브라우저를 사용하여 웹 서핑을 하고 난 이후에 사용하던 컴퓨터의 전원을 끄고 잠들고 나서, 잠시 몇 시간 이후에 다시 컴퓨터를 켜고 이전에 탐색했던 사이트에 확인해야 할 것이 있어서 브라우저를 켜고 이전 페이지로 돌아가기 버튼을 눌렀을 때 그 해당 페이지가 다시 열리도록 지원하는 방법이다.

어플리케이션 캐시란?

  • 우분투에서 파이어폭스 브라우저를 열고 기록에 남아있는 사이트에 다시 방문했을 때 비록 네트워크에 연결되어 있지 않더라도 해당 페이지의 첫 페이지가 열리는 것이 바로 캐시가 사용된 예이다.
  • 캐시를 사용하여 다시 이전 페이지로 접근하려 하더라도, 제대로 페이지가 캐싱되어 있지 않은 경우에는 페이지를 제대로 볼 수 없다.
  • 어플리케이션 캐시의 적용 예로 구글 캘린더에서 사용하는 오프라인 기능이 대표적이다.

어플리케이션 캐시란?

  • HTML4까지만 하더라도 오프라인 작업을 하기 위해서는 사용자가 각 페이지를 일일이 저장해야만 했지만, HTML5에서는 보다 효율적인 방법을 제공해준다.
  • 오프라인 작업을 사용자들이 효율적으로 사용하도록 하기 위해서 HTML5는 어플리케이션 캐시(Offline application cache)를 제공한다.

어플리케이션 캐시 상태

  • 각 어플리케이션 캐시는 상태를 가지는데 이는 캐시의 현재 상태를 나타내준다. 동일한 매니페스트 URI를 공유하는 캐시들은 동일한 캐시 상태를 공유하며 다음의 상태 중 하나이다.
  • UNCACHED : 이 상태에서의 어플리케이션 캐시 객체는 완전히 초기화되지 않았다.
  • IDLE : 어플리케이션 캐시는 지금 업데이트되는 상태에서 미존재 상태이다.
  • CHECKING : 매니페스트는 업데이트를 위해서 확인하는 상태이다.
  • DOWNLOADING : 변경된 리소스 매니페스트로 인하여 리소스들은 캐시에 추가되기 위해 다운로드 되었다.
  • UPDATEREADY : 새로운 업데이트가 다운로드되지만 활성화되지는 않은 경우에는 해당 업데이트 이벤트가 캐시 이벤트 대신 사용된다.
  • OBSOLETE : 어플리케이션 캐시 그룹은 현재 쓸모없는 상태이다.

어플리케이션 캐시의 리소스

  • 어플리케이션 캐시의 리소스 사용은 매니페스트 파일의 URI에 의해서 지정된다. 또 캐시를 구성하는 리소스는 적어도 하나 이상이어야 한다.
  • 여러 개가 존재할 경우에 각각은 URI에 의해서 구분된다. 모든 리소스들은 다음에서 설명하는 엔트리 카테고리 중에 하나가 된다.
  • 마스터 엔트리들
  • 사용자가 방문한 브라우징 컨텍스트가 문서를 포함하고 있으므로, 캐시에 추가되는 리소스들을 의미한다.

어플리케이션 캐시의 리소스

  • 마스터 엔트리들은 (html) 엘리먼트에서 매니페스트 특성을 포함하는 HTML 파일들을 의미한다.
  • entry.html이 매니페스트에 포함되지 않으면 사용자가 entry.html 페이지에 방문할 때 마스터 엔트리로서 어플리케이션 캐시에 추가하게 된다.
  • 명시적 엔트리들
  • 캐시의 매니페스트가 명시적으로 포함하는 리소스들이다.
  • 폴백(Fallback) 엔트리들
  • 폴백(Fallback) 엔트리로 캐시의 매니페스트에서 포함하고 있는 리소스들을 의미한다.

어플리케이션 캐시의 리소스

  • 네트워크 엔트리들
  • NETWORK하에서 나열된 엔트리들을 의미한다. 이는 웹 어플리케이션이 온라인으로 액세스할 수 있으며, 온라인 'whitelist' 내에 존재한다.
  • 네트워크 엔트리의 URI는 캐시 대신 서버로부터 로드된다.

어플리케이션 캐시 동작 방법

  • 어플리케이션 캐시는 HTML 엘리먼트에 매니페스트 특성을 부여하는 것으로 매니페스트 특성 부여는 다음과 같이 명시함으로써 이루어진다.

어플리케이션 캐시와 매니페스트 파일

  • 매니페스트 파일은 어플리케이션 캐시에 다운로드되고 저장되어질 리소스(HTML, JavaScript, CSS, 이미지 파일)들을 기술한다.
  • 웹 페이지가 첫 번째로 로딩된 후 부터는 매니페스트 파일에 기술되어진 리소스 파일들을 웹 서버가 아닌 어플리케이션 캐시에 의해 얻을 수 있게 된다.
  • 각 웹 어플리케이션은 리소스 매니페스트를 가지며, 리소스 매니페스트는 자신만의 캐시를 가진다.

어플리케이션 캐시와 매니페스트 파일

    매니페스트 파일 속성

  • MIME-Type이 'text/cache-manifest'와 같이 제공되어야 한다.
  • 모든 리소스들은 어플리케이션 캐시 매니페스트용 문법을 따르는 MIME 타입을 사용해야 한다.
  • 캐시 매니페스트는 UTF-8 포맷의 텍스트 파일이며 선택적으로 BOM 문자를 포함한다.
  • 첫 번째 줄에는 'CACHE MANIFEST'라는 텍스트를 포함하고 있어야 한다. 두 단어 사이에는 공백이 존재해야 한다.
  • 주석문들은 # 문자로 시작하고 한 줄이어야 한다.
  • URL들은 로컬에 캐시로 다운로드를 원하는 리소스에 대한 파일 경로이다.
  • 경로는 매니페스트 파일에 대한 상대/절대 경로로 표현되어야 한다. CSS에서 사용되는 파일 경로와 유사하다.

  • 매니페스트 파일을 선언한 HTML 파일은 자동적으로 어플리케이션 캐시에 포함된다.
  • 매니페스트 파일을 생성한 후에는 HTML 파일에 선언을 해 주어야 한다. 다음과 같이 (html) 태그에 'manifest' 속성을 추가하면 된다.
  • 어플리케이션 캐시를 생성하는데 매니페스트 파일을 생성과 더불어 선언도 이루어져야만 한다.

어플리케이션 캐시 업데이트 및 확인

  • 사용자는 어플리케이션 캐시가 자동적으로 업데이트되기를 기다리거나 혹은 수동으로 자바스크립트를 이용하여 업데이트할 수 있다.
  • 먼저 브라우저가 매니페스트 특성을 포함하는 문서를 열 때 'window.applicationCache' 객체로 체킹 이벤트를 전송한다.
  • 적합한 HTTP 캐싱 규칙에 맞추어서 매니페스트 파일을 가지고 온다.
  • 만일 매니페스트 파일의 현재의 캐싱된 복사본이 업데이트 된다면, 'noupdate' 이벤트가 'applicationCache' 객체로 전송되고 업데이트 절차는 완료된다.
  • 매니페스트 파일을 변경되면, 매니페스트의 모든 파일들은 적합한 HTTP 캐싱 규칙에 의해서 임시캐시로 저장된다.
  • 각 파일들이 캐시로 저장되면 진행 이벤트가 'applicationCache' 객체로 전송된다.

웹 스토리지(Web Storage) API

  • HTML5를 이용하면 인터넷에 연결되지 않아도 동작이 가능한 독립적인 어플리케이션 앱을 만들 수 있다. HTML5의 오프라인 지원 기능으로 로컬 저장장치에 데이터를 저장할 수 있기 때문이다.
  • 이렇게 저장된 자료는 웹 SQL로 자료를 조회할 수 있어 데이터에 대한 접근성이 향상된다. 웹 워커(Web Worker)를 사용하여 쓰레드 기반으로 동작하는 어플리케이션을 사용할 수 있으며 웹 소켓(Web Socket)을 사용하여 대용량의 데이터를 이전의 버전보다 훨씬 빠르게 처리할 수 있다.

웹 스토리지의 개념

  • 데이터의 저장은 '키'와 '값' 쌍으로 이루어지며, 데이터 조회는 키를 기반으로 이루어진다.
  • 또한 데이터의 지속성을 보장하기 위해서 다음과 같은 두 개의 저장소를 둔다.
  • 영구 저장소(Local Storage)
  • 임시 저장소(Session Storage)
  • 웹 스토리지는 기존 웹 환경에서 브라우저가 제공하고 있으며 사용자들이 손쉽게 접할 수 있는 쿠키(Cookie)와 매우 비슷한 특성을 가지고 있다.

스토리지(Storage) API

  • 스토리지 API는 전형적인 이름/값 쌍 데이터 구조를 가진다. 사용하게 될 가장 일반적인 메소드로 다음과 같은 두 가지가 있다.
  • getItem(name) : 이름과 연관된 값을 리턴하거나 해당 항목이 없을 때는 널(Null)을 반환한다.
  • setItem(name, value) : 이름과 연관된 값을 리턴하거나 해당 항목이 없을 때는 이름/값 쌍을 'localStorage'에 추가하거나 기존 값을 대체한다.
  • 그 외의 API는 다음과 같다.
  • removeItem(name) : 이름/값 쌍이 있으면 'localStorage'에서 이름/값 쌍을 제거하고 이름/값 쌍이 없으면 아무것도 수행하지 않는다.
  • length() : 저장 중인 이름/값 쌍의 총 수를 제공하는 역할을 담당한다.
  • key(index) : 스토리지에 사용된 모든 이름의 배열에서 이름을 리턴한다.
  • clear() : 모든 데이터를 스토리지에서 삭제한다.

웹 워커(Web Worker)

  • 웹 워커를 사용하면 웹 브라우저에서도 데스크탑 어플리케이션과 마찬가지로 멀티태스킹이 지원된다.
  • 웹 브라우저에서 지원하는 멀티태스킹과 완전히 동일하지는 않지만 웹 브라우저의 UI 쓰레드와 완전히 분리된 상황에서 백그라운드(Background)로 동작한다.
  • 웹 워커는 백그라운드로 독립적으로 돌아가는 스크립트를 수행시키는 것과 관련 있는 API에 대해서 정의하고 있는데 여기서 의미하는 독립적으로 그리고 백그라운드로 돌아가는 스크립트는 클릭이나 다른 사용자의 동작에 의한 다른 스크립트에 의해서 발생한 인터럽트로 인하여 방해받지 않는다.

웹 워커 이벤트 루프

  • 웹 워커의 각 'WorkerGlobalScope' 객체는 하나의 이벤트 루프를 가진다.
  • 이 이벤트 루프는 브라우징 컨텍스트(Browsing context)와 관련된 유닛들에 대해서 정의된 것과는 다른 특성을 가지고있으며, 브라우징 컨텍스트 그리고 이 컨텍스트의 태스크 큐(이벤트 포함), 콜백, 네트워킹 액티비티와는 관련을 가지지 않는다.
  • 각 'WorkerGlobalScope' 객체는 클로징 플래그를 가지는데, 이는 초기값으로 'false' 값을 가진다.
  • 이 값은 프로세싱 모델에서 제공하는 알고리즘에 의해서 'true‘로 설정될 수 있다.

웹 워커 이벤트 루프

    워커의 생명주기는 다음과 같다.

  • 워커들은 다른 워커들과 통신을 수행하며, 이를 위해서 메시지 채널과 각각의 'MessagePort' 객체를 통한 브라우징 컨텍스트를 사용한다.
  • 각 'WorkerGlobalScope' 워커의 글로벌 스코프(Global scope)에서는 워커의 포트 리스트를 가진다.
  • 각 'WorkerGlobalScope'는 또한 워커의 또 다른 워커의 리스트를 가진다.
  • 문서(Document) 객체 d가 워커의 문서에 추가될 때, 각 워커에 대한 사용자 에이전트는 'd'를 포함하지는 않고 'd'를 'WorkerGlobalScope' 사용자의 리스트에 추가한다.

    • 웹 워커 사용을 위한 스크립트 작성은 다음과 같이 하도록 한다.

  • 우선 자바스크립트 언어에 맞는 스크립트 실행 환경을 설정한다.
  • 스크립트를 사용하여 소스를 파싱한다. 그리고 코드 엔트리 포인트 리스트를 얻는다.

  • [단계 2] 워커가 공유 워커이면서 만약에 워커의 글로벌 스코프 값이 'SharedWorkerGlobalScope’ 객체라면, 매니페스트의 URL로 구분이 가능한 어플리케이션 캐시들이 존재하는 것이므로, 워커의 글로벌 스코프 값을 적합한 어플리케이션 캐시를 사용하여 설정한다.
  • [단계 3] 비동기 플래그를 설정하여, URL에 의해서 구분되는 리소스를 가진다.
  • [단계 4] 위의 동작이 성공되면, 스크립트 리소스는 UTF-8로 디코딩이 이루어지며, 에러 핸들링도 이에 맞추어 이루어진다.

웹 워커 프로세싱 모델

    브라우저가 워커를 강제로 종료시킬때, 워커의 메인 루프에서 병렬적으로 다음의 스텝을 수행해야만 한다.

  • [스텝 1] 워커의 'WorkerGlobalScope' 객체가 가지고 있는 종료 플래그를 'true'로 설정한다.
  • [스텝 2] 만약에 이벤트 루프의 태스크 큐에 태스크가 존재한다면, 이를 처리하지 않고 버린다.
  • [스텝 3] 워커에서 현재 수행하고 있는 스크립트를 취소한다.
  • [스텝 4] 브라우저는 사용자 요청 기반으로 워커에서 강제 종료 프로세스 모델을 실행시킨다.

웹 워커 함수

  • 웹 워커에서는 중요한 함수가 두 개 존재한다. 이 웹 워커 함수들은 UI Thread와 백그라운드(Background) 사이에서 통신을 담당한다.
  • 함수 'postMessage()'를 사용하여 메시지를 전송하면 먼저 등록해 놓은 'onmessage’이벤트 핸들러로 메시지를 받는다.
  • 그리고 다시 'postMessage()'로 결과를 되돌려 주며, 'onmessage' 이벤트 핸들러로 결과를 받게 된다.
  • 웹 워커의 가장 간단한 사용 방법은 사용자 인터페이스를 통한 인터럽트 없이 태스크를 그냥 수행하는 것이다.

웹 SQL 데이터베이스(Web SQL Database)

  • 웹 SQL 데이터베이스는 브라우저에 키와 값을 저장하던 웹 스토리지를 확장한 개념으로, 데이터베이스를 활용하여 좀 더 다양한 데이터를 저장할 수 있도록 지원한다.
  • 웹 SQL 데이터베이스는 클라이언트인 브라우저에 경량의 관계형 데이터베이스 엔진을 내장하고 있어 SQL로 데이터를 액세스할 수 있는 기능을 제공한다.
  • 따라서 웹 스토리지가 비교적 적은 양의 데이터를 저장하는데 사용되었다면 웹 SQL 데이터베이스는 보다 복잡한 구조로 이루어진 관계형 데이터를 대량으로 저장하는데 사용된다.

웹 SQL 데이터베이스(Web SQL Database)

  • 웹 SQL 데이터베이스는 아이폰이나 안드로이드와 같이 모바일 환경에 많이 채용된 경량의 무료 DB엔진인 SQLite를 기반으로 하고 있다.
  • SQLite는 소스가 완전히 공개된 오픈 소스로 간단한 임베디드 데이터베이스(Embedded Database)에서 많이 사용된다.
  • 하지만 현재 웹 SQL 데이터 베이스는 Firefox 및 IE에서는 지원되지 않고 있으며 웹킷 기반 브라우저(Safari, Chrome 및 안드로이드 브라우저)에서는 이 웹 SQL 데이터베이스를 지원하고 있다.

브라우저별 지원 현황

웹 SQL 데이터베이스의 기본 API

    openDatabase()

  • 'openDatabase()'는 데이터베이스를 열거나 생성하는데 사용된다. 이 API는 총 4개의 필수 파라미터와 1개의 선택 파라미터를 입력으로 받는다.
  • name : 데이터베이스 이름
  • version : 버전 번호
  • displayName : 데이터베이스의 설명
  • estimatedSize : 예상 크기
  • (optional) creationCallback : 데이터베이스 생성시 호출되는 콜백 함수

웹 SQL 데이터베이스의 기본 API

    transaction()

  • 데이터베이스를 열거나 생성한 이후에는 데이터의 안전한 저장 및 처리를 위해 트랜잭션 매커니즘이 필요하다.
  • 웹 SQL 데이터베이스에서 트랜잭션을 위한 API를 일부 제공하고 있는데, 바로 'transaction()'과 'readTransaction()'이다.

웹 SQL 데이터베이스의 기본 API

    executeSql()

  • 'executeSql()'은 실제로 데이터를 실제로 읽고 쓰는 명령을 수행하는 API이다.
  • 이 함수에 파라미터로 전달되는 SQL을 수행하고 파라미터에서 지정한 콜백 메소드에서 그 결과를 확인할 수 있다.
  • 'executeSql()'의 파라미터로 SQL외에 3개의 파라미터가 추가로 존재한다.
